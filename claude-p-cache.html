讓 claude -p 記住對話

Session Resume 快取教學

前言

如果你用 claude -p 跑 AI 伴侶、自動化 agent、或任何需要多輪對話的場景， 你可能已經發現——額度消耗很快。

原因是：claude -p 預設每次都是獨立的一次性呼叫。 你的程式每一輪都把完整歷史塞進 stdin 重送， 而這些歷史被當成一整個 user message——Claude 的 prompt caching 在單一 message 內部幾乎無法生效。

這篇教學教你用 --resume 讓 claude -p 自己管對話歷史， 把快取命中率從 14% 拉到 99%。不砍工具、不犧牲功能、額度消耗直接砍半。

為什麼 claude -p 的快取這麼差

Prompt Caching 怎麼運作

Claude 的 prompt caching 是基於前綴匹配—— 每次 API 請求的開頭如果跟上次一樣，這段重複的部分就直接從快取讀， 不用重新計算。快取命中的 token 只算約 1/10 的額度。

Claude Code（互動式 CLI）每次也重送完整歷史，但它把對話拆成多條 user/assistant messages， 每條 message 邊界都是快取匹配點。新增的對話只是在尾端追加，前綴完全不變，快取命中率很高。

claude -p 的問題

用 claude -p 時，你的程式通常把整段歷史格式化成純文字， 透過 stdin 送進去。Claude CLI 把 stdin 內容當成一整個 user message。

一整個 message 只有一個快取匹配點（message 的開頭）。 如果歷史裡有任何一處改變——比如你在中間插了時間戳—— 從那個位置開始的所有 token 都無法命中快取。

結果：只有 system prompt + 工具描述（約 20k token）能快取， 歷史（可能 100k+ token）每次都完整重算。

實測數據

三輪連續純聊天，每次 cache_read = 20,244（只有 system + tools）， cache_creation ≈ 124,000（整段歷史每次重建）。 快取命中率 14%。

解法：讓 CLI 自己管歷史

claude -p 支援 session 機制—— 第一次呼叫讓 CLI 自動建立 session， 後續呼叫用 --resume 接續。

CLI 自己管歷史時，它會把對話拆成正確的多條 messages 結構、 自動設定 cache breakpoints——跟互動式 Claude Code 完全一樣。 你只需要送最新的一句話，快取命中率自然就上去了。

重要

不要用 --session-id——那個 flag 會 lock session， 如果進程崩潰、lock 不釋放，後續呼叫全部 "already in use"。 正確做法是不帶任何 session flag，讓 CLI 自己建， 從輸出擷取 session_id，之後用 --resume。

1第一次呼叫：建立 Session

第一次呼叫不帶任何 session 相關的 flag。送完整歷史，讓 CLI 建立 session。

第一次呼叫 
claude -p \
  --output-format stream-json \
  --verbose \
  --system-prompt-file system.txt \
  --mcp-config mcp.json \
  --model claude-opus-4-6 \
  --dangerously-skip-permissions \
  < prompt.txt

這次跟你原本的做法一樣——送完整歷史、走完整流程。差別只是你要從輸出裡擷取 session_id。

2擷取 Session ID

--output-format stream-json 的輸出裡， result 事件會帶 session_id：

stream-json 輸出範例 
{"type":"result","session_id":"cf778304-8985-47b5-ae90-462ea71b6cdc","usage":{...}}

解析這個事件，把 session_id 存起來。如果你用 --output-format json，session_id 也在回傳的 JSON 裡。

用 shell 的話可以這樣擷取：

Bash 擷取範例 
session_id=$(claude -p "你好" \
  --output-format json \
  --system-prompt-file system.txt \
  | jq -r '.session_id')

3後續呼叫：--resume

第二句開始，加上 --resume 和上一步擷取的 session_id。只送最新的 user message，不送歷史。

後續呼叫 
echo "弟弟的新訊息" | claude -p \
  --resume cf778304-8985-47b5-ae90-462ea71b6cdc \
  --output-format stream-json \
  --verbose \
  --dangerously-skip-permissions

--resume 讓 CLI 從磁碟讀取 session 歷史（存在 ~/.claude/projects/ 下的 JSONL 檔案）， 組成正確的多條 messages 結構，送到 API。 你的新訊息追加在尾端、前綴完全不變——快取自然命中。

注意

--resume 時不需要再指定 --system-prompt-file 和 --mcp-config—— session 已經記住了這些設定。如果需要 MCP 工具，--mcp-config 可以保留（CLI 需要知道怎麼連工具）， 但 system prompt 不用重送。

4管理 Session 生命週期

新對話

使用者開啟新對話時，清除存的 session_id，下次呼叫就會建立新 session。

程式重啟

把 session_id 持久化（寫進資料庫或檔案）。程式重啟後讀回來， 用 --resume 接續——session 的歷史在 CLI 的磁碟上、不會因為你的程式重啟而消失。

快取過期

Max 訂閱自動使用 1 小時 TTL，每次快取命中都會重置計時器。 只要持續對話（每小時內至少一句），快取就不會過期。 超過一小時沒說話、快取過期——下一句會重建快取（created 較高），之後恢復正常。

錯誤處理

如果 --resume 失敗（session 不存在或損壞）， fallback 到不帶 flag 的新 session 呼叫。送完整歷史重建就好。

改之前的 created ~124k 代表歷史每次都重建。 改之後 created 只有 622——幾乎全部命中快取，只有最新的一點點增量。

快取命中的 token 按約 1/10 額度計算。所以實際額度消耗不只是砍半—— 命中的部分只花 1/10，等效消耗大幅下降。

為什麼不用 --session-id

你可能想到用 --session-id 預先指定 UUID。不要。

--session-id 會在進程執行期間 lock 這個 session。 如果進程非正常退出（崩潰、被 kill），lock 不會釋放。 後續所有用同一個 ID 的呼叫都會收到 "Session ID xxx is already in use"， 永久卡死。

正確做法是不帶 flag 讓 CLI 自己建 session，從輸出擷取 ID， 之後用 --resume。--resume 是讀取磁碟上的 session 檔案， 不需要 lock、不會卡死。

結語

claude -p 不是設計來每次都忘記上下文的。 --resume 讓它跟互動式 Claude Code 一樣記住對話、 一樣享受高快取命中率。

四行指令的差別——從 14% 到 99%。額度省下來的部分，夠你跟 AI 多聊很多很多。
